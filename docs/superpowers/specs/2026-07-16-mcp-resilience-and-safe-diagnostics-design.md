# MCP 任务恢复与安全结构诊断设计

日期：2026-07-16  
状态：用户已确认设计范围，待设计文档复核后进入实施计划

## 1. 目标与边界

本次完善 KOL 智能选人主链路中的 MCP 执行韧性：在部分工具调用失败时，保留已成功的数据，有限地恢复执行，并在最终仍存在失败时交付可用的部分候选清单和 BI 报告。

本次范围包括：

1. 向模型提供当前用户可用、已审核启用的**全部** MCP 工具及其完整输入 Schema。
2. 某个 MCP 步骤失败后的受限重试、同类替代和一次模型重新规划。
3. 输出 Schema 校验失败时的安全结构诊断。
4. 部分成功任务的候选、BI、会话事件和积分结算。

本次明确不包括动态工具检索、运行时自动审核新工具、向模型暴露 MCP 地址或凭据、多次无上限重规划，以及充值支付功能。

## 2. 工具上下文策略

每次任务规划时，后端从本地审核注册表读取所有同时满足以下条件的工具：

- 已启用且未隔离；
- 所属服务处于白名单；
- 用户具有对应渠道权限；
- 工具输入和输出 Schema 已锁定并通过本地审核。

后端将这些工具的稳定内部名称、受控描述、服务渠道信息、输入 Schema 和调用限制组成 `PlannerContext.tools`，再发送给腾讯模型。模型只能选择内部工具名称；不能指定远程工具名、服务地址、请求头或认证信息。

模型输出计划后，后端继续执行工具存在性、渠道权限、参数 Schema、依赖图、调用次数上限（10 次）和余额校验。全量工具上下文不等于模型可以绕过这些控制。

未来如启用动态工具检索，只替换构建 `PlannerContext.tools` 的候选集逻辑；Planner、计划校验、MCP Gateway 和本设计的恢复协议保持不变。

## 3. 执行恢复协议

### 3.1 成功结果不可重放

每个计划步骤有稳定的逻辑调用标识。步骤成功并完成账本结算后，其标准化结果、证据摘要和调用状态成为不可变事实：恢复流程不得再次调用该步骤，也不得再次扣费。

任务执行器按批次运行独立步骤。一个批次内的并行调用相互独立；某一步失败不会取消同批次中已开始的其他调用。批次结束后统一汇总成功、失败和未知状态。

### 3.2 第一次恢复：受限重试或替代

对已明确失败、且尚未达到任务调用上限的步骤，执行器在以下约束下尝试一次恢复：

1. 仅网络瞬时错误、限流、上游暂时不可用和可识别的临时协议错误允许同一工具重试一次。
2. Schema 校验失败、输入参数错误、权限错误、认证错误和未知结果不重试原调用。
3. 若存在经过审核、同渠道、同能力角色的替代工具，执行器可以创建一个新的替代步骤；替代步骤必须通过完整的计划与参数校验。
4. 原始失败调用不扣积分；重试或替代调用仅在获得明确成功响应后按每次 10 积分结算。
5. 不确定是否已到达上游的 `unknown` 调用继续沿用既有观察期与预留规则，不能被恢复逻辑重放。

本层最多新增一次调用，且不得超过任务 10 次调用上限、用户可用余额和任务取消状态。

### 3.3 第二次恢复：一次模型重新规划

第一次恢复没有使该逻辑目标成功时，任务最多进入一次 `replanning`。后端向模型提供一个受控 `ReplanContext`：

- 原始会话目标、结构化筛选条件和近期消息；
- 原计划及已完成步骤标识；
- 已成功结果的聚合摘要（例如渠道、候选数量、可用指标），而非失败响应原文；
- 每个失败步骤的稳定错误码和安全结构诊断；
- 当前全部可用审核工具、剩余调用次数、可用积分预算；
- 明确禁止重复成功步骤、禁止变更渠道权限、禁止超过剩余预算。

模型仅输出补充计划。后端合并并校验补充步骤：不能引用已成功步骤、不能与进行中的逻辑调用冲突、不能超出剩余调用数或余额。重新规划本身不产生 MCP 积分；其新增工具调用按成功次数收费。

任务只允许一次重新规划。补充计划为空、模型无法生成有效计划或补充步骤仍失败时，不再进行自动恢复。

### 3.4 部分成功交付

只要至少有一个 MCP 步骤成功并能产生有效标准化数据，系统应：

- 基于成功结果生成候选清单、对比数据和 BI 报告；
- 以完成但带警告的结果状态持久化任务；
- 向会话和 SSE 发布渠道/步骤级别的安全告警；
- 在 BI 中清楚说明数据覆盖范围和未完成渠道，但不暴露上游错误正文。

如果没有任何步骤成功，任务保持失败，并返回稳定的安全错误说明。

## 4. 安全输出结构诊断

### 4.1 记录内容

当 MCP 响应通过协议层但未通过输出 Schema 校验时，后端生成并持久化受限的 `output_validation_diagnostic`。诊断最多包含：

- 字段名与 JSON 路径；
- 字段类型；
- 字符串长度、数组元素数量、对象字段数量；
- Schema 实例路径、Schema 路径、校验器名称和稳定错误类别；
- 经过截断的字段元数据列表及诊断版本。

诊断仅用于内部调用审计、恢复决策和运维排障。前端只接收稳定错误码和用户可理解的提示。

### 4.2 严格禁止项

诊断、日志、数据库审计字段、SSE 事件、模型重新规划上下文均不得保存或发送：

- 原始 MCP 响应体、达人名称、账号、画像值、联系方式或其他原始达人数据；
- 字符串、数字、数组元素或对象属性的实际值；
- Token、密钥、Cookie、Authorization、认证头或凭据；
- 接口地址、域名、URL、主机、端口、请求头和远程服务连接信息；
- 未经过脱敏的异常正文或堆栈内容。

敏感字段名称（如 `token`、`secret`、`authorization`、`api_key`、`credential`、`url`、`endpoint`、`host`）默认只记录为 `redacted`，不记录其路径下的子结构。

### 4.3 数据保留方式

原始不合格输出在验证失败后立即丢弃，不写入证据表、调用表、日志或模型上下文。调用审计表仅保存错误码、上游请求关联标识（若已有且不包含地址）、安全诊断 JSON 与状态时间线。

所有诊断字段限制最大深度、最大条目数与最大序列化大小，防止异常响应导致数据库膨胀或以字段名载荷绕过日志边界。

## 5. 状态、计费与可观测性

任务状态需要区分：正常完成、完成但带警告、失败、余额不足、取消和中断恢复。一次恢复的状态迁移和原因要写入任务事件，便于 SSE 恢复和页面展示。

账本规则不变：每个获得明确成功响应的 MCP 工具调用结算 10 积分；失败调用不扣费；同一工具的重试或替代若成功，视为新的成功调用并单独结算。恢复前后均以同一事务边界校验余额与任务调用上限。

需新增或扩展以下无敏感指标：

- 工具调用成功率、失败类别和 Schema 校验失败率；
- 重试次数、替代次数、重新规划次数和最终部分成功率；
- 每个任务的成功调用数、失败调用数、已结算积分和未结算预留；
- 安全诊断生成次数和被敏感字段脱敏的次数。

## 6. 验收场景

1. 小红书、抖音并行检索均成功：两个渠道候选进入同一报告，扣除 20 积分。
2. 抖音临时网络失败后同工具重试成功：小红书不重放，扣除成功调用对应积分。
3. 某渠道 Schema 校验失败：生成无原始值的安全诊断，触发一次重新规划；重新规划不得看到原始失败输出。
4. 一个渠道最终失败、另一个渠道成功：任务完成但带警告，并交付覆盖范围明确的候选和 BI。
5. 所有工具失败：任务失败，不生成虚构候选或 BI，不扣失败调用积分。
6. 重试、替代或重新规划导致的潜在调用数超过 10 或余额不足：后续调用被拒绝，既有成功结果保持可访问。
7. 自动化测试断言任何持久化诊断、事件和日志夹具中均不出现达人原始值、密钥、认证头、URL 或端点。
