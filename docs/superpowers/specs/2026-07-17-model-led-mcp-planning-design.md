# 模型主导的品牌/KOL 动态分析设计

## 1. 目标

将分析任务的“分析对象、证据类型、MCP 工具选择和调用参数”交给腾讯模型
DeepSeek-V4-Pro 统一规划。系统不再通过固定关键词直接决定 `brand`、`kol` 或
`hybrid`，而是把用户输入、会话上下文以及全部已授权 MCP 工具能力交给模型，
由模型选择能够回答问题的最小且充分的工具计划。

用户关于品牌或 KOL 的自然语言问题都应进入同一条真实数据链路：

```text
用户输入
  -> 中性上下文构建
  -> DeepSeek-V4-Pro 生成结构化 ToolPlan
  -> 安全/权限/Schema/预算校验
  -> DataTap MCP 分批调用
  -> 空结果自动重新规划（有预算时）
  -> 候选清单 + BI 报告 + 会话分析
```

## 2. 非目标

- 本阶段不实现动态工具检索；每次规划仍向模型提供全部已授权、可用的 MCP
  工具定义和输入/输出 Schema。
- 不改变 MCP 的计费规则：每个成功响应的工具调用扣 10 积分，失败调用不扣费，
  同一工具重复调用按次数计费。
- 不让模型接触 MCP 地址、令牌、数据库凭据或未经清洗的原始敏感数据。
- 不允许模型直接生成“真实数据”；所有报表数值必须来自成功的 MCP 响应。

## 3. 核心决策

### 3.1 ContextBuilder 只负责提供事实，不做业务范围裁决

`ContextBuilder.build()` 仍然汇总：

- 当前会话、历史消息和用户最新问题；
- 用户可用的平台权限；
- 全部已授权 MCP 工具的名称、用途、输入 Schema、输出 Schema、平台能力和
  失败/空结果语义；
- 已有的 MCP 证据摘要（仅保留字段名、类型、长度、Schema 路径和统计摘要）；
- 当前积分余额、单任务调用上限和并发限制。

它不再把关键词分类结果作为 `PlannerContext.analysis_scope` 的强制前置条件。
旧的规则分类器只能作为模型规划失败时的安全兜底，不能覆盖成功的模型计划。

### 3.2 ToolPlan 是唯一的执行依据

模型返回 JSON，不返回可执行代码。建议结构：

```json
{
  "primary_intent": "brand|kol|hybrid",
  "objectives": ["品牌声量趋势", "用户情感趋势"],
  "steps": [
    {
      "step_id": "brand-trend-xhs",
      "tool_name": "datatap.insight.social.statistic.trend.v1",
      "evidence_kind": "brand",
      "platform": "xiaohongshu",
      "depends_on": [],
      "arguments": {
        "brand": "科颜氏",
        "period": "最近3个月"
      },
      "reason": "获取小红书品牌声量和趋势"
    }
  ]
}
```

执行前从 `steps[*].evidence_kind` 推导最终 `analysis_scope`。新任务必须至少包含
一个 `kol` 步骤，因此新任务只有两种范围：

- 只有 `kol`：`kol`；
- 同时存在 `brand` 和 `kol`：`hybrid`。

纯 `brand` 仅用于兼容历史任务，不再由新规划生成。无论用户询问品牌趋势、
情感、话题还是达人筛选，模型都必须规划至少一个匹配相关 KOL 的真实 MCP 步骤；
涉及品牌指标时还应规划品牌分析步骤。

因此下面的用户问题不会被硬编码规则阻断，而是由模型根据工具能力补齐所需
证据：

| 用户问题 | 模型主意图 | 允许的最终范围 | 计划示例 |
| --- | --- | --- | --- |
| 分析科颜氏最近三个月声量和情感趋势 | `brand` | `hybrid` | 品牌趋势、情感、匹配/活跃达人补充 |
| 找适合科颜氏的达人 | `kol` | `kol` 或 `hybrid` | 达人搜索、画像、受众匹配，并可补品牌上下文 |
| 分析品牌趋势，并找出相关达人 | `hybrid` | `hybrid` | 品牌趋势 + 各平台达人搜索/画像 |

`primary_intent` 是解释性元数据；实际执行和报表以 `steps` 为准，避免模型
声明了范围却没有对应证据时出现契约不一致。

### 3.3 安全校验仍由后端确定性完成

模型拥有规划权，但没有执行权。后端在运行 MCP 前必须校验：

1. 工具名称属于已授权 MCP 工具目录；
2. 平台属于用户授权范围，且工具能力声明支持该平台；
3. 参数符合工具输入 JSON Schema，移除未声明字段并补齐安全默认值；
4. 依赖图无环，单任务最多 10 个调用，批次并发遵守网关限制；
5. 参数不包含令牌、地址、SQL、任意 URL 或其他敏感字段；
6. 证据类型与工具输出契约一致；
7. 预算/积分不足时在调用前中止，并将可读错误写入会话事件。

原有 `EVIDENCE_SCOPE_NOT_COVERED` 不再依据一个预先硬编码的 scope 触发。
后端始终校验新计划至少含一个 `kol` 步骤；当模型声明品牌分析意图时还必须含
一个 `brand` 步骤。缺少证据时返回可重新规划的结构化错误；空计划、非法工具
或 Schema 错误仍然是不可执行错误。模型调用失败时，旧规则分类器仅用于生成
安全兜底计划，不能覆盖成功的模型计划。

### 3.4 空结果与失败后的重新规划

- MCP 调用成功但返回空列表，不视为系统异常；记录工具、平台、空结果摘要，
  不写入达人原始数据。
- 若仍有预算和未尝试的合适工具/平台，向模型发送“已成功但为空”的摘要，
  让模型生成不超过剩余预算的新计划；不得重复相同的工具+参数组合超过一次。
- MCP 调用失败按错误类别记录：超时、上游 4xx/5xx、鉴权、Schema、限流、
  网络和未知错误。失败调用不扣积分。
- 达到重规划次数/预算上限后，保留成功证据，生成带“数据不足/部分渠道失败”
  标记的 BI 报告，不阻止页面展示可用内容，也不伪造指标。

### 3.5 持久化与多轮会话

每轮任务保存：原始用户问题、模型规划请求摘要、模型返回的 ToolPlan、后端
校验后的最终计划、推导出的 `analysis_scope`、MCP 调用事件、失败摘要、候选
版本和 BI 报告版本。下一轮会话读取历史消息和上一轮结构化摘要，模型可在此基础
上增量规划；不会复用其他会话的 BI 数据。

## 4. 数据安全与日志

成功证据进入归一化/脱敏层后才可用于候选和 BI。失败日志只记录：

- task/session/tool/platform；
- 错误类别、HTTP 状态或 MCP 错误码；
- 字段名、字段类型、长度和 Schema 校验路径；
- 重试/重规划次数、耗时和扣费结果。

禁止记录达人原始内容、完整响应体、令牌、MCP 地址和用户隐私信息。

## 5. 前端可见行为

任务状态事件由后端真实事件驱动：接收请求、模型规划、计划校验、MCP 查询、
重新规划、AI 汇总、BI 生成、完成或失败。失败事件必须带用户可理解的分类和
下一步建议。即使部分渠道失败，只要存在成功证据，右侧 BI 和候选清单仍显示，
并标注不可用渠道。

## 6. 验收标准

1. 不调用规则分类器时，三类示例问题都能由模型生成不同的结构化计划；每个
   新计划至少有 KOL 证据，品牌指标问题同时有品牌证据，最终 scope 由计划证据
   类型推导为 `kol` 或 `hybrid`。
2. 模型可从完整 MCP 目录中选择品牌趋势、情感、达人搜索、达人画像等工具，
   计划通过后端安全校验后才执行。
3. 计划缺少必要证据、含未授权工具或非法参数时，任务不会调用 MCP，并能返回
   可重规划的错误事件。
4. 一个品牌分析问题即使没有显式出现“达人”，也必须补充匹配 KOL 证据；若
   MCP 返回空结果，页面展示真实的“数据不足”而非假数据。
5. 所有 MCP 失败均可在后端日志中按类别检索，且不泄露敏感信息。
6. 多轮会话能引用上一轮摘要重新规划，BI、候选和扣费记录按 task/session 隔离。
