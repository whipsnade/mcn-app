# 品牌与 KOL 动态分析设计规格

## 目标

系统不再把每个会话都当作 KOL 候选筛选。用户问题进入后，后端先识别分析范围和目标，再将完整的已审核 DataTap 工具目录、字段契约和会话上下文交给 DeepSeek-V4-Pro 规划真实 MCP 调用；最终按真实证据生成 KOL 候选、品牌分析或二者联合的 BI 报告。

## 已确认规则

- 不实现外部动态工具检索；每轮规划都传入当前用户可用的全部已审核工具。
- 未选择渠道时，查询当前用户已授权的全部平台；明确选择渠道时，仅查询所选平台。
- MCP 调用仍按每个成功工具响应 10 积分计费；失败不计费。
- 任何报告字段只能来自已成功且通过输出校验的 MCP 证据；不得生成模拟数值。
- MCP 成功但无匹配数据时，任务可以完成并生成空结果报告，界面明确显示“暂无匹配数据”。
- MCP 失败、规划失败或模型失败时，任务事件和会话消息必须显示实际阶段及安全错误摘要。

## 分析范围与路由

后端从当前用户问题和会话条件计算路由提示：

- `brand`：品牌声量、提及趋势、情感分布、热词、受众画像等品牌问题；
- `kol`：达人检索、活跃达人、粉丝画像、平台比较、候选排序等 KOL 问题；
- `hybrid`：品牌问题同时要求活跃 KOL、品牌提及达人或平台候选时，要求两类证据。

路由提示不是模型结果，也不替模型选择工具。它用于约束规划目标、要求至少覆盖对应证据类型，并作为后端校验的一部分。DeepSeek 仍从完整工具目录中选择具体工具和参数。

## 组件边界

### PlannerContext

新增安全的 `analysis_scope`、`analysis_objectives` 和 `requested_period` 投影字段。`tools` 保留全部已审核工具，但每个工具必须带有能力描述、服务、输入 Schema 和输出 Schema。上下文中不出现密钥、URL、原始 MCP 载荷或达人原始数据。

### Planner

规划 Prompt 明确要求：

1. 先判断范围是 `brand`、`kol` 还是 `hybrid`；
2. 只选择能覆盖用户目标字段的工具；
3. 需要跨平台时为每个可用平台规划对应工具；
4. brand 任务不能仅返回 KOL 搜索工具；hybrid 任务必须包含品牌证据和 KOL 证据；
5. 每个步骤声明 `evidence_kind`（`brand` 或 `kol`）和 `evidence_goal`；
6. 后端校验步骤数量、工具权限、Schema、渠道和范围覆盖。

后端只补全会话明确要求的稳定筛选参数，不替模型捏造品牌指标或趋势数据。

### MCP 证据

新增安全的品牌证据规范化适配器。适配器只投影 BI 白名单字段，例如平台、日期或周期、声量、曝光、互动、情感计数、热词、受众分布及来源调用引用；不保存原始载荷、URL、密钥、达人账号原始详情。

KOL 适配器继续生成候选规范化记录。两类证据在同一任务中独立聚合，允许同一平台同时贡献品牌和 KOL 数据。

### ReportingService

报告生成按任务范围分流：

- `kol`：生成候选版本和 KOL BI；
- `brand`：不要求候选版本，直接由品牌证据生成品牌 BI；
- `hybrid`：同时生成候选版本（允许为 0）和品牌 BI。

报告保存的 `chart_data_json` 增加 `analysis_scope`、`brand_analytics`、`kol_analytics`、`data_availability` 和 `warnings`。旧报告继续通过兼容读取返回原有结构。

### 空结果与错误

- 所有 MCP 已成功但品牌或 KOL 证据为空：任务状态为 `completed_with_warnings`，报告可读取，字段使用 `available=false`、`coverage=0`，并显示中文说明；
- 某些工具失败但仍有可用证据：保留可用结果，任务为 `completed_with_warnings`，报告列出失败渠道；
- 没有任何成功证据：任务仍生成明确的失败阶段消息和安全错误，不创建伪造报告；
- 诊断日志仅记录字段名、字段类型、长度和 Schema 路径，不记录原始数据、密钥或接口地址。

## 前端交付

右侧 BI 依据 `analysis_scope` 显示：

- 品牌分析：品牌声量趋势、平台分布、情感趋势、热词、受众画像；
- KOL 分析：候选清单、评分、平台分布、筛选条件和导出；
- 联合分析：品牌 BI 与候选 KOL 两个区块同时显示。

当字段不可用时保留报表框，但展示“暂无真实 MCP 数据”，不使用占位数值。会话阶段根据实际事件显示规划、品牌 MCP 查询、KOL MCP 查询、数据归一化、BI 汇总和错误信息。

## 验证标准

1. “分析科颜氏最近 3 个月在各平台的声量变化和用户情感趋势”会规划品牌证据，不会只规划 KOL 搜索；
2. “找最近 30 天活跃达人”会规划 KOL 证据；
3. “分析品牌声量并找出相关活跃达人”会规划两类证据；
4. 跨平台任务的成功调用按平台分别记录并合并，单个平台无数据不会覆盖其他平台；
5. MCP 成功空结果不再触发 `no_successful_tool_evidence`；
6. 全部测试、类型检查、lint 和构建通过。
